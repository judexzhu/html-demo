Konnectivity in HyperShift
How the control plane tunnels traffic to worker nodes when they live in separate networks.
⚠ HyperShift / Hosted Control Planes. In HyperShift the control plane runs in a management cluster (different network) from the worker nodes. Konnectivity provides a reverse tunnel so the API server, operators, and proxy sidecars can reach kubelets, services, and endpoints on the data plane.

Traffic Flows
1
API Server → Kubelet
1. API server initiates request (e.g. kubectl logs)
2. Egress selector routes cluster traffic to konnectivity-server-local :8090
3. Server selects tunnel to target worker node agent
4. Agent forwards to local kubelet on port 10250
2
Aggregated API (e.g. OpenShift API)
1. Service (no selector) + Endpoints created in hosted cluster
2. API server routes APIService request through tunnel
3. Control plane konnectivity-agent intercepts (--agent-identifiers)
4. Routes back to actual aggregated API server in CP namespace
3
Proxy Sidecar Traffic
1. Operator initiates connection (e.g. catalog-operator)
2. Traffic goes to local socks5-proxy or https-proxy sidecar
3. Sidecar connects to konnectivity-server-local :8090
4. Server routes through agent tunnel to data plane
4
Agent Registration (reverse)
1. Worker konnectivity-agent dials konnectivity-server :8091 (outbound)
2. Server responds with unique ID + total server count
3. Agent keeps dialing until connected to all servers (HA)
4. Persistent tunnel maintained for control plane push traffic
